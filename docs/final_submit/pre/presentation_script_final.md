## 幻灯片 2：动机与问题陈述

**视觉建议：** 分屏对比。左侧：传统文字游戏（僵化决策树、"我不理解"错误提示、有限选项）。右侧：StoryWeaver（动态分支、自然语言输入、无限可能）。中间用红色箭头标注 "幻觉问题"。

**幻灯片内容：**
* **传统文字冒险的局限：**
  * 僵化的硬编码决策树
  * 玩家输入超出预设 → "我不理解"
  * 长会话中 AI 遗忘早期情节（幻觉）
* **StoryWeaver 的突破：**
  * 自然语言输入，无限动态情节
  * 知识图谱锚定世界状态，消除矛盾
  * 本地 NLU + 云端/本地 LLM 混合架构

**演讲者备注（约 50 秒）：**
传统文字冒险游戏依赖硬编码的决策树——玩家只能从预设选项中选择，一旦输入超出脚本范围，游戏就会说"我不理解"。而现代基于 LLM 的游戏虽然灵活，却面临更严重的问题：AI 会遗忘早期事件、复活已死亡的角色、甚至让一个角色同时出现在两个地方。这就是所谓的"幻觉"问题。

StoryWeaver 的解决方案是：用结构化的知识图谱作为"基本事实来源"。
**举个极端的例子**：如果玩家在第 2 回合杀死了地精，但在第 10 回合 LLM 却试图生成“地精向你打招呼”。在我们的系统中，这种逻辑矛盾会被知识图谱立刻捕获并拦截。我们不是把原始对话历史塞给 LLM，而是先用本地 NLU 模型理解意图，再用图谱的结构化状态来约束生成。这样既保留了 LLM 的创造力，又确保了叙事的绝对一致性。

## 幻灯片 3：系统架构

**视觉建议：** 清晰的高层架构图（与 README 一致）。数据流从左到右：玩家输入 → NLU 模块 → 游戏引擎编排器 → 故事/选项生成（LLM）↔ 知识图谱。顶部标注 Streamlit 前端。用不同颜色区分本地模块（绿色）和 API 模块（蓝色）。

**幻灯片内容：**
* **前端：** Streamlit UI — 聊天界面 + PyVis 交互式 KG 可视化 + NLU 调试面板
* **NLU（本地）：** DistilBERT（意图分类）+ spaCy（实体抽取）+ fastcoref（共指消解）+ DistilRoBERTa（情感分析）
* **NLG（三种模式）：** API（Mimo v2 Flash）/ 本地（Qwen3-4B via llama.cpp）/ 混合
* **世界状态：** NetworkX MultiDiGraph + 双通道冲突检测 + 重要性评分
* **工程鲁棒性：** 惰性加载 + 透明兜底（模型失败时自动降级为规则匹配）

**演讲者备注（约 60 秒）：**
这是 StoryWeaver 的整体架构。系统分为四层：

最上层是 Streamlit 前端，不仅提供游戏聊天界面，还包含基于 PyVis 的交互式知识图谱可视化——节点按实体类型着色（人物用青色 #00f0ff，地点用绿色 #39ff14），以及 NLU 调试面板，方便我们观察 AI 的底层决策过程。

第二层是本地 NLU 流水线，包含四个模块：fastcoref 解决代词歧义，微调的 DistilBERT 将玩家意图分类为 8 个类别，spaCy 抽取游戏实体，以及基于 DistilRoBERTa 的情感分析器识别 7 种 Ekman 情绪。所有模型本地运行，快速且零成本。

第三层是 NLG 生成模块，通过 HybridClientManager 路由，支持三种模式：纯 API 调用 Mimo v2 Flash、纯本地运行 Qwen3-4B、以及混合模式——创意故事生成走本地，结构化选项生成走 API。

最底层是基于 NetworkX MultiDiGraph 的知识图谱，配合双通道冲突检测。整个系统实现了惰性加载和透明兜底：每个 NLU 模块的 load() 方法最多重试 3 次，失败后自动降级为关键词规则匹配，确保 100% 可用性。

## 幻灯片 4：NLU 流水线 — 理解玩家意图

**视觉建议：** 流程图展示示例句子的转换过程。"I attack the goblin with it" → fastcoref 消解 "it" → "the goblin" → DistilBERT 分类为 "attack" → spaCy 抽取实体 "goblin"。用彩色标签标注每个模块的输出。

**幻灯片内容：**
* **共指消解（fastcoref）：** "I attack it" → "I attack the goblin"，失败时降级为代词规则映射
* **意图分类（DistilBERT）：** 8 类意图，兜底使用 KEYWORD_MAP 评分匹配
* **实体抽取（spaCy NER + 启发式）：** 生物/地点/物品/魔法词集，含模糊匹配消歧
* **情感分析（DistilRoBERTa）：** 7 种 Ekman 情绪（愤怒/厌恶/恐惧/喜悦/悲伤/惊讶/中性），注入故事生成提示词

**演讲者备注（约 55 秒）：**
让我们深入 NLU 流水线。当玩家输入 "I attack it with the sword" 时，系统依次执行四步：

首先，fastcoref 根据最近的故事情境，将 "it" 消解为具体的实体，比如 "the goblin"。如果模型加载失败，系统使用预定义的代词映射规则进行规则消解。

接着，消解后的文本送入我们专门微调的 DistilBERT 模型，将意图分类为 8 个类别之一——这里是"攻击"。如果模型不可用，系统自动切换到 KEYWORD_MAP 关键词评分匹配。

然后，spaCy 的 NER 结合自定义名词短语启发式方法，抽取出具体的游戏实体。我们预定义了生物、地点、物品、魔法四类词集，并与 KG 实体进行模糊匹配消歧。

最后，情感分析器使用 j-hartmann/emotion-english-distilroberta-base 模型识别 7 种 Ekman 情绪，情绪结果直接注入 StoryGenerator 的 continue_story 方法，让叙述者根据玩家情绪调整故事基调。

## 幻灯片 5：知识图谱 — 维持叙事一致性的核心

**视觉建议：** 图谱网络可视化截图（PyVis 交互式效果）。高亮展示：节点属性（状态、位置、关系）、"检测到冲突！"警告框、分层摘要的生成过程。

**幻灯片内容：**
* **重要性评分公式：** 0.3×度数中心性 + 0.3×近期性(0.95^n) + 0.2×提及频率 + 0.2×玩家提及
* **分层摘要：** Core(≥0.6) / Secondary(≥0.3) / Background(<0.3)，优化 LLM 上下文窗口
* **双通道冲突检测：**
  * 通道 1 — 确定性规则：EXCLUSIVE_PAIRS（alive/dead, ally/enemy）+ 时序因果检查
  * 通道 2 — LLM 仲裁：置信度分级（>0.75 接受 / 0.45-0.74 延迟 / <0.45 忽略）
* **冲突解决：** KeepLatestResolver（按 last_confirmed_turn 保留最新）+ LLMArbitrateResolver

**演讲者备注（约 75 秒）：**
知识图谱是 StoryWeaver 的"记忆中枢"。我们的方案不是将整个聊天历史塞给 LLM——而是在每回合后抽取实体和关系，更新图谱。

图谱采用 NetworkX MultiDiGraph 实现。我们为每个实体设计了重要性评分系统：重要性 = 0.3×度数中心性 + 0.3×近期性（0.95 的未提及回合数次方衰减）+ 0.2×提及频率 + 0.2×玩家提及频率。未被提及的实体重要性会逐回合衰减，关系置信度同样按衰减因子降低，低于阈值的自动修剪。

基于重要性评分，to_summary() 方法生成分层摘要：Core 实体（≥0.6）提供完整属性、状态、情绪和近期历史；Secondary 实体（≥0.3）提供基本信息；Background 实体（<0.3）仅保留名称和类型。这为 LLM 提供了紧凑而精准的上下文。

当新关系被抽取时，双通道冲突检测立即启动。通道一是确定性规则：EXCLUSIVE_PAIRS 定义了互斥关系对如 alive/dead、ally/enemy，以及"死亡-活跃"冲突检测——已死亡实体不能再创建新关系。同时进行时序因果检查，识别因果倒置。通道二是 LLM 仲裁：LLM 分析世界状态摘要和新文本，识别深层逻辑矛盾，并按置信度分级处理——高于 0.75 直接接受，0.45 到 0.74 延迟处理，低于 0.45 忽略。

冲突解决由 KeepLatestResolver 和 LLMArbitrateResolver 协同完成：前者根据 last_confirmed_turn 自动保留最新关系，后者查询 LLM 决定具体操作（保留新/旧、删除关系、更新实体等）。

## 幻灯片 6：NLG 与动态故事生成

**视觉建议：** Streamlit 界面截图，高亮生成的故事文本和三个动态选项（带风险等级标签：低/中/高）。右侧标注提示词模板的结构。

**幻灯片内容：**
* **上下文感知生成：** 玩家意图 + KG 分层摘要 + 7 种情绪基调 + 历史
* **动态选项生成：** 每回合 3 个选择，风险分级（低/中/高），llm_client.chat_json() 结构化输出
* **三种 NLG 模式：**
  * API：Mimo v2 Flash，最高质量
  * 本地：Qwen3-4B via llama.cpp，隐私优先
  * 混合：故事→本地，选项/关系抽取→API
* **LLM 无关架构：** 通过 HybridClientManager 路由，可无缝切换

**演讲者备注（约 55 秒）：**
在故事生成方面，我们设计了结构化的提示词模板，将玩家的分类意图、情绪基调、知识图谱分层摘要和历史记录融合为一个紧凑的上下文包。这确保 LLM 生成的叙事续写严格遵守已建立的世界规则。

每回合结束时，选项生成模块通过 chat_json 结构化输出为玩家创建三个不同的选择，按风险等级分类：低风险（安全探索）、中风险（常规行动）、高风险（戏剧性转折）。即使玩家不想输入自定义响应，这也能保持游戏的参与感。

我们的架构是 LLM 无关的——通过 HybridClientManager 的 get_client_for_task 方法路由。你可以用 Mimo API 获得最高质量，也可以用本地 Qwen3-4B 保护隐私，或者用混合模式：创意故事生成走本地模型，结构化的选项生成和关系抽取走 API。三种模式通过 config.py 的 NLG_MODE 一键切换。

## 幻灯片 7：评估与定量结果

**视觉建议：** 仪表盘风格布局。左侧：KG On/Off 对比柱状图（一致性、实体覆盖率、Distinct-2、Self-BLEU）。右侧：LLM-as-Judge 评分仪表盘（9.12/10）。

**幻灯片内容：**
* **KG On vs Off 基准测试：**
  * 一致性比率：0% → **100%** (+100%)
  * 实体覆盖率：0.79 → **1.00** (+26%)
  * Distinct-2：0.64 → **0.74** (+15%)
  * Self-BLEU：0.36 → **0.22** (-38%，越低越好)
* **LLM-as-Judge 评分：** 7.50 → **9.12 / 10** (+21%)
* **评估指标：** Distinct-N（词汇多样性）、Self-BLEU（会话内多样性）、Entity Coverage（KG 实体利用）、Consistency Rate（无冲突回合百分比）

**演讲者备注（约 60 秒）：**
为确保学术严谨性，我们实现了全面的评估套件。由于交互式叙事没有单一的"正确答案"，我们采用无参考指标和对比实验。

我们进行了最关键的 KG On/Off 基准测试。结果显示：启用知识图谱后，叙事一致性从 0% 飙升至 100%——这意味着完全消除了逻辑矛盾。实体覆盖率提升 26%，证明生成的故事真正利用了图谱中的世界细节。Distinct-2 提升 15%，Self-BLEU 降低 38%，说明 LLM 不再陷入重复循环。

此外，我们使用 LLM-as-Judge 方法进行类人质量评估，整体评分从 7.50 提升至 9.12 分。这些数据有力证明了知识图谱对生成质量的实质性贡献。
